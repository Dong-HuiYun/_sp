# 生產者消費者模擬 — Producer-Consumer

示範在多執行緒環境下，如何用 `std::mutex` 與 `std::condition_variable` 協調生產與消費的速度差，安全地共用一個有界緩衝區。

---

## 問題背景

生產者不斷產生資料，消費者不斷取用資料，兩者速度不一定相同，因此需要一個緩衝區作為中繼站。但緩衝區容量有限，必須處理兩種邊界情況：

- **緩衝區滿了**：生產者不能繼續放，必須等消費者取走後才能放
- **緩衝區空了**：消費者不能繼續取，必須等生產者放入後才能取

如果不處理這兩種情況，輕則讀到垃圾資料，重則程式直接崩潰。

---

## 解決方案

使用一個 `mutex` 加上兩個 `condition_variable`：

| 同步原語 | 角色 | 等待方 | 通知方 |
|----------|------|--------|--------|
| `mutex` | 保護 queue，防止同時讀寫 | — | — |
| `not_full` | 代表「緩衝區有空位」這個條件 | 生產者（滿時等待） | 消費者（取走後通知） |
| `not_empty` | 代表「緩衝區有資料」這個條件 | 消費者（空時等待） | 生產者（放入後通知） |

生產者與消費者透過互相通知形成自動協調的節奏，不快也不慢。

---

## 程式設計

### BoundedBuffer 類別

| 成員 | 說明 |
|------|------|
| `queue_` | 實際儲存資料的佇列，受 `mutex_` 保護 |
| `capacity_` | 緩衝區容量上限，本程式設為 3 |
| `mutex_` | 保護 `queue_` 的互斥鎖 |
| `not_full_` | 生產者用來等待空位的條件變數 |
| `not_empty_` | 消費者用來等待資料的條件變數 |
| `produce(item)` | 放入資料，滿則阻塞等待 |
| `consume()` | 取出資料，空則阻塞等待 |

### 執行緒配置

| 執行緒 | 數量 | 行為 |
|--------|------|------|
| 生產者 | 1 | 每 50ms 生產一個 item，共生產 10 個 |
| 消費者 | 2 | 每 120ms 消費一個 item，各消費 5 個 |

生產者比消費者快（50ms vs 120ms），緩衝區會先被填滿，生產者因此必須停下來等待，展示 `not_full` 的作用。

### 一次完整的生產流程

```
1. 等待 50ms（模擬生產耗時）
2. 取得 mutex
3. 檢查 queue 是否已滿
   → 若滿：暫時釋放 mutex，進入睡眠，等 not_full 通知
   → 若有空位：繼續
4. 將 item 放入 queue
5. 通知 not_empty（告知消費者有東西可取）
6. 釋放 mutex
```

消費者流程為完全對稱的鏡像，方向相反。

---

## 等待時為何要釋放 mutex

這是最容易忽略的細節。若生產者拿著 mutex 在那邊等空位，消費者就永遠無法進入 `consume()` 取走資料，也就永遠無法製造空位，兩邊互相等待，形成死鎖。

`condition_variable::wait()` 的設計解決了這個問題：進入等待的瞬間自動釋放 mutex，讓其他執行緒可以進來操作；被通知喚醒後再重新取得 mutex，才繼續執行。

---

## 虛假喚醒（Spurious Wakeup）

`condition_variable::wait()` 在極少數情況下會被作業系統無故喚醒，稱為虛假喚醒。因此等待時必須搭配條件判斷（predicate），被喚醒後再次確認條件是否真正成立，若不成立就繼續等待：

```cpp
// 錯誤：可能虛假喚醒後直接繼續執行
not_empty_.wait(lock);

// 正確：條件不成立就繼續等待
not_empty_.wait(lock, [this] { return !queue_.empty(); });
```

## 編譯與執行

進入`wsl`環境

```bash

g++ Producer_consumer.cpp -o Producer_consumer
./Producer_consumer

```

## 預期輸出

```
=== 生產者消費者模擬 ===
緩衝區容量 = 3，生產 10 個 item，由 2 個消費者分攤消費

  [Producer] 生產 item=  1  (緩衝區: 1/3)
  [Producer] 生產 item=  2  (緩衝區: 2/3)
  [Consumer] 消費 item=  1  (緩衝區: 1/3)
  ...
  [Consumer] 消費 item= 10  (緩衝區: 0/3)

[Done] 所有 item 已生產並消費完畢。
```

緩衝區數字不會超過 3，也不會出現負數，代表協調機制正常運作。
