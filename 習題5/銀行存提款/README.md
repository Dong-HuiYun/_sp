# 銀行存提款模擬 — Bank Simulation

示範在多執行緒環境下，如何用 `std::mutex` 保護共享資料，避免競態條件（Race Condition）導致帳戶餘額錯誤。

---

## 問題背景

多個執行緒同時對同一個變數執行 `balance += amount`，看似一行程式，CPU 實際拆成三步：

1. **LOAD**：從記憶體讀取 balance 到暫存器
2. **ADD**：暫存器值加上 amount
3. **STORE**：將結果寫回記憶體

當兩個執行緒的這三步互相交錯執行，其中一個執行緒的寫入結果就會被另一個覆蓋，導致金額憑空消失。

```
初始 balance = 1000

執行緒 A：LOAD  → 讀到 1000
執行緒 B：LOAD  → 讀到 1000  ← 讀到舊值，不知道 A 的存在
執行緒 A：ADD   → 1001
執行緒 B：ADD   → 1001
執行緒 A：STORE → balance = 1001
執行緒 B：STORE → balance = 1001  ← 覆蓋 A！兩次存款只加了一次
```

---

## 解決方案

使用 `std::mutex` 搭配 `std::lock_guard`，將讀取、計算、寫回這三步包成一個不可分割的原子操作。

```
執行緒 A：取得鎖 → LOAD → ADD → STORE → 釋放鎖
執行緒 B：          等待...等待...取得鎖 → LOAD → ADD → STORE → 釋放鎖
```

執行緒 B 被強制等到 A 完整跑完三步之後才能開始，不再有交錯的可能。

## 程式設計

### BankAccount 類別

| 成員 | 說明 |
|------|------|
| `balance_` | 帳戶餘額，所有執行緒共享的資料 |
| `mutex_` | 保護 `balance_` 的互斥鎖 |
| `deposit(amount)` | 加鎖後執行 `balance_ += amount` |
| `withdraw(amount)` | 加鎖後檢查餘額，足夠才執行 `balance_ -= amount` |

### 執行緒配置

| 執行緒 | 數量 | 操作 | 次數 |
|--------|------|------|------|
| 存款執行緒 | 2 | `deposit(1)` | 各 100,000 次 |
| 提款執行緒 | 2 | `withdraw(1)` | 各 100,000 次 |

總存入與總提出相等，初始餘額與最終餘額應完全一致。

## 重要觀念

**`std::lock_guard` 的 RAII 機制**
建構時自動加鎖，離開作用域時自動解鎖，即使函式中途拋出例外也不會忘記解鎖，比手動呼叫 `lock()` / `unlock()` 更安全。

**移除 mutex 會發生什麼**
若將 `lock_guard` 那行刪除，400,000 次操作中幾乎必定發生多次覆蓋，最終餘額會是一個錯誤的數字，且每次執行結果都不同，難以重現與除錯。
